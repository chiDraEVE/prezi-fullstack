In der IT hört man diesen Begriff immer häufiger, den man jedoch leider nicht konkret definiert hat. In der Regel meint man mit Fullstack-Developer einen Webentwickler, der sowohl am Frontend als auch am Backend arbeitet. Dabei spricht man beim Frontend über all das, was im Browser passiert, und beim Backend, was auf dem Server passiert. Bisher hat man Aufgaben in diesen Bereichen auf unterschiedliche Personen verteilt. Im Backend wird teilweise sogar die Entwicklung der Datenbank auf einen oder mehrere Entwickler aufgeteilt.

Jetzt seht ihr auf der Folie noch ein paar Punkte stehen, mit denen Programmierneulinge erst mal gar nichts anfangen können. In anderen YouTube-Videos zu Full-Stack-Development hört man oft "Die und die Technologien müsst ihr lernen, um Full-Stack-Developer zu sein". Meine Punkte hier sind auch nur ein Beispiel eines Full-Stacks. Da gibt es ganz viele Möglichkeiten, dazu später mehr.

Damit ihr aber jeder ein besseres Bild davon habt, was Frontend und Backend ist, erkläre ich das an einem kleinen Beispiel.

Weitere essentielle Technologien

  • Web-Server
  • Agile-Development
  • CI / CD
  • Build-Tool
  • Tests
  • Versionskontrolle

Für einen - ich sag jetzt mal - "kompletten" Full-Stack-Developer reichen Frontend und Backend alleine nicht aus. Er muss sich auch mit folgenden Themen Beschäftigen

  • Web-Server - damit er weiß wie diesen für sein Programm anpassen muss
  • Agile Development und Continious Integration bzw. Continious Delivery - da Entwicklungszyklen immer kürzer werden und Änderungen bereits früh ausgeliefert werden
  • Build-Tools - für das automatische Einbindung von Abhängigkeiten und das Ausliefern als Pakete
  • Tests und Versionskontrolle

Für die letzten beiden Punkte zeige ich euch ein Beispiel, damit man sich etwas darunter vorstellen kann. Übrigens werde ich auf diesem Kanal alle Punkte, die ich hier erwähne, in zukünftigen Videos im Detail behandeln. Tests in IntelliJ zeigen

Hier haben wie zwei Tests. Beim ersten Test wird geprüft, ob die Aufgabe "Task 1" in der Liste auftaucht, wenn ich die entsprechende Funktion dafür benutze.
Mit Hilfe dieses Tests habe ich herausgefunden, dass der Code, wie ich ihn aus dem Buch kopiert habe nicht mehr funktioniert. Die Funktion konnte nämlich keine neuen Aufgaben hinzufügen, wenn die Liste zuvor leer war. Also musste ich den Code erweitern. Zeige entsprechende Zeile Die Variable, die mir die ID-Nummer der letzten Aufgabe geben soll, war früher bei einer leeren Liste vordefiniert. In der aktuellen Version tritt ein Fehler auf, den ich abfangen musste.

So Sachen passieren ständig. Deswegen ist es extrem wichtig im ganzem Entwicklungsprozess diese Tests durchzuführen, um zu gewährleisten, dass der bereits erstellte Code immer noch das tut, was er tun soll.
Leider gibt es viele, die denken Tests sind Zeitverschwendung. Bspw. sagte mir mal jemand: "Für Tests zahlt keine Kunde". Dabei sollte es eigentlich heißen, keine Kunde zahlt für eine Software ohne Tests. Denn je größer ein Projekt wird, welches keine Tests hat, umso schwieriger wird es dieses zu warten und umso mehr Zeit und Geld muss man für die Weiterentwicklung reinstecken. Die Qualität vom Code ist dementsprechend auch miserabel.

Solltet ihr als mal in einer Firma oder Team sein, die Tests für unnötig hält,... Dann lauft! So schnell ihr könnt. Irgendwann kommt man nämlich an einem Punkt, wo das Projekt nicht mehr vorankommt und Entwickler, Chefs und Kunden drehen durch.

Der zweite Tests übrigens, prüft ob sich die Menge einer Liste um eins vermindert, wenn man die erste Aufgabe entfernt.

Zum Thema Versionskontrolle möchte ich mich auch kurz fassen und das wesentliche erklären: Damit möchte man quasi die unterschiedlichen Entwicklungsstände abspeichern. Während man am Code arbeitet, ist dieser in der Regel nicht kompilierbar. Man würde also nicht die Seite im Browser sehen, wie ich sie vorhin gezeigt habe. Hat man aber nun schon mehrere Stunden oder Tage an seinem Code gearbeitet und man soll sich umgehend um ein dringendes Problem kümmern, so wäre es katasrophal, müsste man alle Änderungen zurücknehmen bis der Code wieder kompiliert. Git Log zeigen Also erstellt man eine sogenannte Branch, zu deutsch Zweig, und macht dann auf ihr seine Änderungen. Sollte nun solch dringender Fall auftreten, checkt man die letzte kompilierbare Version aus und kümmert sich um die Problemlösung. Dann wird das ganze committed und gepusht, das nennt man halt so. Danach kann man sich wieder seiner alten Aufgabe widmen und kehrt auf seine "Branch" zurück.

Diese Branch wird dann wieder mit der letzten kompilierbaren Version "verbunden". Das nennt man dann "Merge". Das Programm ist nun erweitert worden und befindet sich in einem ausführbarem Zustand. In der Regel macht man für jede Anforderung eine neue Branch, die man dann wieder zusammenfügt. Dann ist auch wieder wichtig, dass die Tests geschrieben und alle erfüllt wurden. Schließlich wurde ja Code geändert und man muss ja irgendwie sicher sein, dass der Code noch immer tut, was er tun soll. Das ganze ist natürlich noch viel wichtiger, wenn man im Team arbeitet.

Beispiele von Stacks

  • LAMP - Linux, Apache, MySQL, PHP
  • MEAN - MongoDB, ExpressJS, AngularJS, NodeJS
  • Django - Python Web Framework
  • Ruby on Rails

Stack ist nicht gleich Stacks. Auf dieser Folie stehen ein paar Beispiele von gängigen Stacks. Damit möchte ich eigentlich auch nur sagen, dass es verschiedene Möglichkeiten gibt Technologien zu kombinieren. Stack bedeutet nämlich genau das. Ein Stapel an Technologien für die ganze Bandbreite der Webentwicklung.

Es ist übrigens auch möglich verschiedene Stacks zu kombinieren. Beispielsweise als Micro-Services. Dabei teilt man ein größeres Software-Projekt in mehrere Teilprojekte. Für jedes kann einen eigenen Stack benutzen und mit den anderen über eine REST-Schnittstelle kommunizieren zu lassen. Micro-Services ist übrigens auch ein Buzzword, nach dem in letzter Zeit immer mehr gefragt wird. Auch dazu möchte in Zukunft ein Video machen

Vorteile
  • weniger Kommunikation nötig
  • einheitlicher Stil
  • Einsatz in Front-End oder Back-End
Nachteile
  • generalisiertes statt Expertenwissen

Kommen wir nun zu der letzten Folie, über die Vor- und Nachteile eines Fullstack-Entwicklers.
Auf der einen Seite haben wir, dass weniger Kommunikation nötig ist, da der Fullstack sich nicht mit Frontend bzw. Backend absprechen muss. Was auch zu einem einheitlichem Stil in der Programmierung führt.
Und selbstverständlich kann man den Fullstack in einem auch als Frontend- oder Backend-Entwickler einsetzen.

Auf der anderen Seite haben wir, dass der Fullstack eben ein eher generalisiertes Wissen hat, da er sich in so viele Technologien einarbeiten muss. Für spezielles Wissen besteht der Bedarf an Experten.
